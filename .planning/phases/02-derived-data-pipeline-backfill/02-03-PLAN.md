<objective>
Build the index layer that makes the derived output consumable and cache-bustable: `stations.json` (the marker/registry manifest generated from the Phase-1 no-splice registry, gated to stations with >=3 qualifying years of real daily data) and `manifest.json` (content-hashed derived filenames + per-station high-water marks). Content hashing is what turns each nightly redeploy into a delta for returning visitors (DATA-04) and what Phase 8's cron reads to know where to resume (DATA-07 bookkeeping).

Purpose: Phase 3's client fetches `stations.json` to place markers and `manifest.json` to resolve each station's immutable derived URL. This plan produces both, offline-testably, using Phase-1 primitives (`parseStationsBody`, registry) and Node `crypto`.

Output: `manifest.ts` (`contentHash` + `updateManifest`), `stations.ts` (`buildStationsJson`), and tests proving the qualifying-years filter, decommissioned retention, and hash-changes-iff-bytes-change delta property.
</objective>

# Facts / policy (02-RESEARCH)
- `start`/`ending` are INTEGER years; `start` is a year but daily data may begin later — filter on ACTUAL daily data (>=3 qualifying years), NOT on `start`.
- Keep decommissioned stations (`ending != null`) that meet the >=3-qualifying-years bar (locked decision: retained for historical windows).
- 518 stations are day-servable (sj+sk); 347 active — the filter runs over the derived/raw data actually produced.
- Content-hash: `crypto.createHash('sha256')` over the derived file bytes, truncated (e.g. first 8-12 hex chars) → `derived/{station}.{hash}.json`.
- manifest high-water marks: per-station `{ from, to, lastFetched }` (the resume anchor; the on-disk high-water helper is in Plan 02's rawstore — this plan owns the manifest JSON that records it).
